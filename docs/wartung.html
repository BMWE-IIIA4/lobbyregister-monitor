Lobbyregister-Monitor · Interne Dokumentation

Wichtig: Echte Zugangsdaten sind ausschließlich als GitHub Secrets hinterlegt. Niemals in den Code eintragen.

1. Systemübersicht

Der Lobbyregister-Monitor ruft täglich inkrementell Stellungnahmen ab:

  • Nur neue Registereinträge werden von der API geladen (bereits bekannte werden übersprungen)
  • KI-Prüfung (Gemini 3.1 Flash Lite) mit lokalem Cache – bereits geprüfte Einträge werden nicht erneut verarbeitet
  • Budgetlimit: max. 100 KI-Calls pro Durchlauf
  • Übersichtsseite auf GitHub Pages, sortiert nach Upload-Datum
  • Ampelfarben zeigen Zeitverzug zwischen Stellungnahme-Datum und Upload an
  • Filterung nach Regelungsvorhaben und Institutionen (kombinierbar)

Zusätzlich jeden Montag: Wochenmail (nur bei automatischem Trigger) und Selbsttest.

Ablauf: täglich um 01:00 Uhr MEZ (= 00:00 UTC).

Sommerzeit: GitHub Actions kennt keine Berliner Zeit. Im Sommer (MESZ = UTC+2) läuft der Workflow um 02:00 Uhr statt 01:00 Uhr. Dies ist technisch bedingt und akzeptiert.

Admin-Passwort ändern

Das Passwort des Admin-Panels liegt nie im Klartext im Code, sondern als Hash im Secret ADMIN_PASSWORD_HASH. Um das Passwort zu ändern:

  1. Browser-Console (F12) öffnen, allow pasting eingeben
  2. Funktion ausführen: async function h(p){const d=(new TextEncoder).encode(p);const b=await crypto.subtle.digest('SHA-256',d);return [...new Uint8Array(b)].map(x=>x.toString(16).padStart(2,'0')).join('')} await h("deinNeuesPasswort")
  3. Kopierten Hash als Secret ADMIN_PASSWORD_HASH speichern
  4. Alternativ: die Datei hash_generator.html lokal öffnen

4. Ablauf eines täglichen Durchlaufs

  1. Cache laden – data.json, gemini_cache.json, run_history.json aus GitHub Actions Cache
  2. fetch_and_build.py – Registernummern laden, nur neue einzeln abrufen, mit vorherigen Daten mergen, initiales HTML bauen
  3. gemini_enrich.py – Neue Einträge per KI prüfen (max. 100/Run), gecachte überspringen, finales HTML mit Ampelfarben und Institutionsfilter bauen
  4. inject_admin_hash.py – Passwort-Hash aus Secret in admin.html einsetzen
  5. save_run_log.py – Durchlauf-Statistiken an run_history.json anhängen
  6. Deploy – GitHub Pages veröffentlichen
  7. Montags (nur automatisch): send_email.py (Wochenmail) + health_check.py (Selbsttest)

5. Manueller Start

Repository → Actions → „Lobbyregister Monitor“ → Run workflow

  • force_email: false (Standard) – Workflow läuft komplett, aber ohne Mail-Versand. Für Tests geeignet.
  • force_email: true – Mail wird zusätzlich versendet. Achtung: Empfänger werden benachrichtigt.

Alternativ: Admin-Panel unter admin.html öffnen (passwortgeschützt).

6. Fehlersuche

Seite nicht aktualisiert

  1. GitHub Actions öffnen
  2. Fehlgeschlagenen Job anklicken und Logs lesen
  3. Häufig: API-Key abgelaufen, GitHub Pages nicht aktiviert, Cache-Problem

Sortierung falsch / Einträge unter falschem Datum

Wenn Einträge unter dem falschen Tagesdatum erscheinen: GitHub Actions Cache löschen (Repository → Actions → Caches) und Workflow neu starten. Der Cache enthält dann keine alten Daten mehr und wird frisch aufgebaut.

Gemini fehlgeschlagen

Erkennbar am gelben Warnbanner auf der Webseite.

  • Quota: 500 req/Tag – Caching und Budgetlimit (100/Run) reduzieren den Verbrauch. Ausstehende Prüfungen werden automatisch nachgeholt.
  • Cache gelöscht: Nach Cache-Löschung müssen alle Einträge neu geprüft werden (>100). Das holt sich über mehrere Tage automatisch auf.
  • Key abgelaufen: Neuen Key auf aistudio.google.com erstellen und als Secret speichern
  • Modell geändert: GEMINI_MODEL in gemini_enrich.py und health_check.py anpassen

Admin-Panel zeigt falsches Passwort

Nach Änderung des Secrets ADMIN_PASSWORD_HASH muss ein neuer Workflow-Run starten, damit der neue Hash in admin.html injiziert wird.
